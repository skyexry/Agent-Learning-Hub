用 DeepSeek 写一个最小 Agent

六步,从「调一次 LLM」到「跑通一个带保险的 agent loop」。每步:可运行代码 + 一句原理 + 一个常见坑。DeepSeek 兼容 OpenAI 接口,所以用 openai 库,只改 base_url;模型 deepseek-chat。

# 环境准备 / setup(绕开 conda,用 venv)
python3 -m venv .venv && source .venv/bin/activate

1

普通对话Plain chat

原理: messages 是 role/content 列表;回复从 choices[0].message.content 取。DeepSeek 客户端 = OpenAI 客户端 + 改 base_url。

import os
from dotenv import load_dotenv
from openai import OpenAI

load_dotenv()  # 从 .env 读 DEEPSEEK_API_KEY

client = OpenAI(
    api_key=os.environ["DEEPSEEK_API_KEY"],
    base_url="https://api.deepseek.com",   # 指向 DeepSeek
)
MODEL = "deepseek-chat"

resp = client.chat.completions.create(
    model=MODEL,
    messages=[{"role": "user", "content": "用一句话解释什么是 agent loop"}],
)
print(resp.choices[0].message.content)
⚠坑: key 走 .env,别 export 在命令行(会进 history、易被截图)。.env 格式 KEY=值,等号两边无空格、不加引号。
2

结构化 JSON 输出Structured JSON

原理: 用 response_format={"type":"json_object"} 强制合法 JSON,再 json.loads 解析。

import json

resp = client.chat.completions.create(
    model=MODEL,
    messages=[{"role": "user", "content":
        "给我《盗梦空间》信息,以 json 返回,"
        "字段:title, year(整数), director"}],
    response_format={"type": "json_object"},
)
data = json.loads(resp.choices[0].message.content)
print(data["year"])   # 2010
⚠DeepSeek 专属坑: 用 JSON 模式时,提示词里必须出现 "json" 字样并说明字段,否则直接报错。生产仍要 try/except 包住解析(token 截断 → 半截 JSON)。
3

定义工具 + schemaDefine tools

原理: OpenAI 风格接口要手写 JSON schema(不像 Gemini 自动从 docstring 生成)。description 就是给模型的说明书,决定它何时调、传什么参数。

def calculator(expression: str) -> float:
    return eval(expression, {"__builtins__": {}}, {})  # 仅演示

def read_file(path: str) -> str:
    with open(path, encoding="utf-8") as f:
        return f.read()

TOOLS = {"calculator": calculator, "read_file": read_file}

TOOL_SCHEMAS = [{
    "type": "function",
    "function": {
        "name": "calculator",
        "description": "计算一个数学表达式并返回结果",
        "parameters": {
            "type": "object",
            "properties": {"expression": {
                "type": "string",
                "description": "合法算术表达式,如 (128 + 64) * 3"}},
            "required": ["expression"],
        },
    },
}]  # read_file 同理,略
⚠坑: description 含糊,模型就乱调或不调。每个参数的 description 要讲清,required 列出必填项。
4

解析 tool callParse tool call

原理: 模型不执行函数,只返回意图 message.tool_calls。执行权在你手里 = act 入口。注意 arguments 是 JSON 字符串,要 json.loads。

resp = client.chat.completions.create(
    model=MODEL,
    messages=[{"role": "user", "content": "帮我算 (128 + 64) * 3"}],
    tools=TOOL_SCHEMAS,
)
msg = resp.choices[0].message
if msg.tool_calls:
    for tc in msg.tool_calls:
        print(tc.function.name)                       # "calculator"
        print(json.loads(tc.function.arguments))     # {'expression': ...}
⚠坑: tool_calls 可能为 None(模型直接回答)或含多个(并行)。要遍历 + 判空,别写死取第一个。
5

执行工具 + 喂回结果Execute & feed back

原理: 一次工具调用 = 至少两次 API 调用。模型无记忆,每次带完整 messages;assistant 的 tool_calls 消息与 role:"tool" 结果消息必须靠 tool_call_id 配对。

messages = [{"role": "user", "content": "帮我算 (128 + 64) * 3"}]

resp = client.chat.completions.create(..., messages=messages, tools=TOOL_SCHEMAS)
msg = resp.choices[0].message
messages.append(msg)                       # assistant 回复入历史

for tc in msg.tool_calls:
    args = json.loads(tc.function.arguments)
    result = TOOLS[tc.function.name](**args)   # 真正执行
    messages.append({
        "role": "tool",
        "tool_call_id": tc.id,             # 必须与 call 对上
        "content": str(result),
    })

final = client.chat.completions.create(..., messages=messages, tools=TOOL_SCHEMAS)
print(final.choices[0].message.content)   # "结果是 576"
⚠坑: ① 忘了把 assistant 的 tool_calls 消息也 append 回历史、只加结果 → 报错。② tool_call_id 配错或漏写 → 报错。一问一答必须成对。
6

加固 agent loopHarden the loop

原理: 套进 while,循环到模型不再发 tool_calls 为止。四道保险:最大步数、未知工具、单工具超时、异常不炸循环。

import concurrent.futures

def run_agent(user_input, max_steps=8, tool_timeout=10):
    messages = [{"role": "user", "content": user_input}]

    for step in range(max_steps):           # 保险1: 最大步数
        resp = client.chat.completions.create(
            model=MODEL, messages=messages, tools=TOOL_SCHEMAS)
        msg = resp.choices[0].message
        messages.append(msg)

        if not msg.tool_calls:           # 无调用 = 最终答案
            return msg.content

        for tc in msg.tool_calls:
            try:
                fn = TOOLS.get(tc.function.name)
                if fn is None:           # 保险2: 幻觉工具
                    raise ValueError(f"未知工具: {tc.function.name}")
                args = json.loads(tc.function.arguments)
                with concurrent.futures.ThreadPoolExecutor() as ex:
                    fut = ex.submit(fn, **args)
                    result = fut.result(timeout=tool_timeout)  # 保险3: 超时
                content = str(result)
            except Exception as e:        # 保险4: 错误喂回,不炸循环
                content = f"ERROR: {e}"

            messages.append({"role": "tool",
                "tool_call_id": tc.id, "content": content})

    return "达到最大步数仍未完成"
★关键设计: 工具报错时把错误喂回模型而非直接抛出 —— 模型看到 ERROR: ... 常能自己改参数重试。这正是 agent 比固定 workflow 更鲁棒的地方。
✓实跑输出: run_agent("先算 (128+64)*3,再加 100") → 模型连调两次 calculator:先得 576,喂回后再算 576+100,最终返回 676。多步循环 + 状态传递跑通。

三家 API 迁移

骨架完全一样,差异只在 Step 4–5 的字段名。换字段即可迁移:
  • DeepSeek / OpenAI: tool_calls / role:"tool" + tool_call_id;arguments 是 JSON 字符串
  • Gemini: function_call / function_response;args 是 dict,工具 schema 自动从 docstring 生成
  • Claude: tool_use / tool_result
